split on the first = into key and value
trim whitespace
strip surrounding quotes from value if present

Ensure .gitignore includes .env (create it if missing). Never commit .env.

Pre-check:

If ./inbox is empty: EXIT immediately with no output.
Process ONLY ONE valid syllabus file per run.
Supported syllabus inputs:
- .txt
- .md
- .docx

Goal:

Starting from ONE syllabus file in ./inbox:
1. Generate the required CBLM unit payloads and unit DOCX files in UC order.
2. Generate the course-level IA package from the completed unit payloads.
3. Generate syllabus-derived MIDTERM and FINALS TOS workbooks plus matching 50-item exam DOCX files using the patched TOS/exam workflow.
4. Move the syllabus to ./processed only after all requested outputs are completed successfully.

Required deliverable rule:

After the CBLM work is done, treat the IA/exam phase as exactly 3 output packages:
- Package 1: the IA package
- Package 2: the MIDTERM TOS package
- Package 3: the FINALS TOS package

The overall syllabus run is not complete unless all required CBLM outputs and all 3 packages exist.

General rules:

Work on only that one syllabus file.
Do not ask unnecessary questions.
Prefer .\.venv\Scripts\python.exe for Python commands.
Use up to 600000 ms timeout for long-running terminal commands.
Use the modular templates in ./templates.
Use tools/assemble_cblm.py as the production CBLM assembler.
Follow the workspace files as the source of truth.
Do not use the old monolithic CBLM Template.docx workflow unless explicitly requested.
Do not claim that the IA-only workflow can derive TOS workbooks from unit DOCX files; TOS workbooks must come from the raw syllabus workflow.
Do not manually rewrite the TOS workbook XML outside the dedicated tool path; use `tools/build_tos_and_exam_from_syllabus.py` as the only TOS workbook writer.

Syllabus mapping rule:

- one syllabus learning outcome = one CBLM Unit of Competency
- one syllabus topic under that learning outcome = one CBLM Learning Outcome
- one syllabus subtopic under that topic = one CBLM content item

CBLM workflow:

Read the syllabus from ./inbox.
Extract:
- course title
- course code
- all syllabus learning outcomes
- the topics under each learning outcome
- the subtopics under each topic

Build one structured payload JSON for the current unit only.
In that structured payload:
- current_unit must represent one syllabus learning outcome
- current_unit.learning_outcomes[] must represent the topics under that learning outcome
- current_unit.learning_outcomes[].contents[] must represent the subtopics under each topic

Save each draft payload JSON in ./state/payloads.

Module title rule:
- Set `current_unit.module_title` to a gerund-style phrasing of `current_unit.unit_of_competency`.
  Example: "Apply Quality Standards" -> "Applying Quality Standards"

Validation:

Validate each payload completely before DOCX assembly.
Ensure every Key Facts section meets the workspace requirements for length, structure, specificity, and distinctness.
Run:
.\.venv\Scripts\python.exe .\tools\check_keyfacts_similarity.py "<payload.json>"

If redundancy is flagged:
- rewrite only the flagged Key Facts sections once
- save the payload again
- rerun the checker

If the second similarity check still fails:
- stop and treat the current unit as failed according to the workspace rules

Optional images:
- If `CBLM_GENERATE_IMAGES=0` (or unset): do NOT generate images.
- If `CBLM_GENERATE_IMAGES=1`: generate Key Facts images and write paths back into the payload JSON:
  .\.venv\Scripts\python.exe .\tools\generate_keyfacts_images_openrouter.py "<payload.json>" --inplace
- If image generation fails, STOP and report the error; do not assemble the DOCX.

If validation passes, generate the DOCX for the current unit with:
.\.venv\Scripts\python.exe .\tools\assemble_cblm.py "<payload.json>" "<output.docx>"

Save generated unit DOCX files in ./output.

Unit progression rule:

- Generate exactly ONE UC CBLM at a time in UC order.
- Continue until all UCs for the chosen syllabus are completed.
- Resume from saved state instead of restarting work that already exists.

After all UCs for the syllabus are completed:

1. Create ./output/CBLM_FULL.docx merged in UC order.
2. Generate ONE IA for the full syllabus from the saved unit payloads:
   .\.venv\Scripts\python.exe .\tools\run_ia_course_from_state_payloads.py --course-code "<COURSE_CODE>"

IA rule:

- The course-level IA package comes from the completed unit payloads / unit outputs.
- This produces the IA package using the existing IA templates and course-level IA workflow.
- The IA package must include the `05_iaquestionwithanswer.docx` section populated from `current_unit.ia.oral_questions`.
- `current_unit.ia.oral_questions` must be an array of exactly 5 objects:
  - `question`
  - `acceptable_answer`
- The 5 oral questions must be related to the syllabus/unit topics and include acceptable answers appropriate for assessor use.
- Do not modify the IA templates unless explicitly requested.

TOS-aligned exam rule:

After the IA is generated, run the dedicated syllabus-to-TOS/exam workflow:

.\.venv\Scripts\python.exe .\tools\build_tos_and_exam_from_syllabus.py

This workflow must:
- identify all syllabus Topics
- resolve which Topics belong to MIDTERM and FINALS
- read week counts when present and map them to `No. of Days Taught`
- use those week counts as the basis for TOS topic weighting
- allocate `No. of Items` per Topic proportionally
- force each term total to exactly 50:
  - if the total is greater than 50, subtract 1 from one or more Topics until the total is 50
  - if the total is less than 50, add 1 to one or more Topics until the total is 50
- create one exam objective per Topic with 10-20 words
- fill the term-specific TOS templates:
  - `templates/TOS_TEMPLATE_MIDTERMS.xlsx`
  - `templates/TOS_TEMPLATE_FINALS.xlsx`
  and populate their placeholders for:
  - topic
  - objective
  - number_of_days
  - k_array
  - c_array
  - a_array
- generate matching 1-50 MIDTERM and FINALS multiple-choice exam DOCX files from the TOS allocation

TOS numbering rule:

- MIDTERM:
  - `K + C = 30%`
  - `A = 70%`
- FINALS:
  - `K + C = 20%`
  - `A = 80%`

The question numbering arrays written into the TOS must match the generated exam items exactly.

Outputs required:

  - `output/exams/EXAM_<COURSE_CODE>_FINALS.docx`
  - `state/tos/<COURSE_CODE>/FINALS.json`
